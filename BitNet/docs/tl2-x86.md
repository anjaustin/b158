# TL2 on x86

This guide covers the end-to-end x86 workflow for running the TL2 LUT kernel path with BitNet.

## When to use TL2

- Use `TL2` on x86 when you want the LUT-based kernel path.
- Start from the bf16 Hugging Face checkpoint, not from a prepacked `i2_s` GGUF repo.
- For BitNet-b1.58-2B-4T, TL2 currently reduces model size and improves decode throughput, while I2_S remains faster on prompt processing in the current single-thread measurements.

## Prerequisites

- x86-64 CPU with AVX2
- Python environment with the repo requirements installed
- A clean buildable `3rdparty/llama.cpp`

## 1. Download the bf16 source model

```bash
huggingface-cli download microsoft/bitnet-b1.58-2B-4T-bf16 --local-dir models/BitNet-b1.58-2B-4T-bf16
```

## 2. Generate TL2 kernels for the 2B model

BitNet-b1.58-2B-4T uses these TL2 kernel shapes on x86:

```text
[2560, 6912]
[6912, 2560]
[2560, 2560]
[640, 2560]
```

Generate the corresponding TL2 code:

```bash
python utils/codegen_tl2.py --model BitNet-b1.58-2B-4T --BM 160,384,160,160 --BK 96,96,96,96 --bm 32,32,32,32
```

## 3. Convert bf16 weights to TL2 GGUF

```bash
python utils/convert-hf-to-gguf-bitnet.py models/BitNet-b1.58-2B-4T-bf16 --outtype tl2 --quant-embd --outfile models/BitNet-b1.58-2B-4T-bf16/ggml-model-tl2.gguf
```

This produces:

```text
models/BitNet-b1.58-2B-4T-bf16/ggml-model-tl2.gguf
```

## 4. Rebuild llama.cpp cleanly

If you changed TL2 codegen, generated headers, or `GGML_BITNET_X86_TL2` build flags, do a full clean rebuild so `ggml.c` and the LUT objects are in sync:

```bash
cd 3rdparty/llama.cpp
make clean
make llama-cli llama-bench -j$(nproc)
```

## 5. Smoke test inference

```bash
./3rdparty/llama.cpp/llama-cli -m models/BitNet-b1.58-2B-4T-bf16/ggml-model-tl2.gguf -p "Hi" -n 32 -t 4 --no-warmup
```

## 6. Benchmark against I2_S

```bash
./3rdparty/llama.cpp/llama-bench -m models/BitNet-b1.58-2B-4T-gguf/ggml-model-i2_s.gguf -t 1 -p 512 -n 128
./3rdparty/llama.cpp/llama-bench -m models/BitNet-b1.58-2B-4T-bf16/ggml-model-tl2.gguf -t 1 -p 512 -n 128
```

Current single-thread reference numbers from this repo state:

| Format | Model Size | `pp512` | `tg128` |
| --- | ---: | ---: | ---: |
| I2_S | 1.71 GiB | 11.03 tok/s | 5.89 tok/s |
| TL2 | 1.02 GiB | 9.93 tok/s | 6.63 tok/s |

## What the profiler says

Recent function- and shape-level profiling on the 2B model changed the optimization picture:

- The x86 `TL2` path is already doing the important ternary work.
- `softmax`, `RoPE`, and `RMSNorm` are small compared with matrix multiplication time.
- The remaining large costs are in non-ternary fallback matmuls, not in the ternary LUT kernels.

Named residual hotspots:

1. `result_output` / logits projection
   - `src/llama.cpp:15537`
   - shape family: `2560 x 128256`
   - path: generic `f16 x f32`
2. `kq` / attention score matmul
   - `src/llama.cpp:9822` and `src/llama.cpp:15468` via `llm_build_kv(...)`
   - shape family: `128 x {32,64,96,512}`
   - path: generic `f16 x f32`
3. `kqv` / value aggregation matmul
   - `src/llama.cpp:9865` and `src/llama.cpp:15468` via `llm_build_kv(...)`
   - shape family: `{32,64,96} x 128`
   - path: generic `f16 x f32`

Practical takeaway:

- If your goal is better end-to-end TL2 speed, the next wins are more likely in the logits path and KV-cache attention kernels.
- Further chasing ternary attention-score softmax or RoPE tweaks is unlikely to move total runtime much on this model.
- In other words, the current bottleneck is the non-ternary edge compute around TL2, not TL2 itself.

## Common pitfalls

- Converting from the prepacked `*-gguf` repo instead of the `*-bf16` checkpoint
- Rebuilding only part of `3rdparty/llama.cpp` after changing TL2 codegen
- Comparing TL2 and I2_S with different thread counts or prompt/generation lengths
